# Retrospective — verify-stage-gate-execution / stage I

> Generated 2026-06-21. Stage completed in commits 351f3e3, 3ce11bd, f72cb46, 32c1e30.

## Friction observed

1. **A slice checkpoint asserted a precise mechanical count the approved
   design predicate could not produce (9 vs 6).** The planner wrote
   `tasks.md` task 1.4 as "exactly 9 Check 5 violations," carried from the
   slice plan. But the body-aware D6 predicate, implemented exactly as
   approved, matched only **6** — because `research`/`plan`/`slices` carry the
   gate-trapping `agent:`+`subtask:` pairing yet never name `AskUserQuestion`
   inline; they reach the gate *transitively* via the `qrspi-workflow`
   choreography. The implementer correctly **stopped and surfaced** the
   discrepancy instead of silently ticking — but that was the implementer's
   good judgement, not a rule the stage-I prompts require. A less careful run
   could have ticked the box at 6, or "fixed" the count to force a match,
   hiding a real design under-coverage.

2. **The root under-coverage was a stage-D blind spot surfacing at I.** D6's
   predicate was specified against the *inline* manifestation of the invariant
   only. The fact that a stage command can invoke a gate tool *transitively*
   through a shared skill was not considered when the predicate was pinned —
   so the guard, as designed, would have under-covered the very bug class it
   exists to catch. It took an implementation run to expose it. The designer
   prompt has several "verify-before-recommending" rules but none that says:
   when you design a self-referential static check, enumerate both the inline
   and the delegated/transitive forms of the target.

3. **A human-only verification task (`5.3`) had no marker.** `tasks.md` 5.3 is
   a manual dogfood (invoke a stage command, confirm the `AskUserQuestion`
   gate fires under the real plugin path) — something a subagent cannot do.
   The implementer correctly left it unticked, but only because the
   orchestrator explicitly told it to; nothing in `tasks.template.md` or the
   implementer prompt distinguishes "agent-executable" from "human-run" tasks.
   Without a convention, such a task either gets attempted-and-failed by the
   subagent, or it silently blocks the "all boxes green" signal forever.

4. **Pre-existing working-tree modifications created reconciliation
   overhead.** Four command files (`questions`/`research`/`slices`/`structure`)
   were already modified in the working tree when stage I began. During Slice 2
   the implementer reported some commands as "already converted," and the
   orchestrator had to diff against HEAD to confirm nothing unrelated was being
   folded into the slice commit. No harm landed (explicit-path staging
   prevented sweeping), but the prompts don't tell the implementer to check for
   and surface pre-existing in-scope edits at the first slice.
