## Archive command flow — detailed

### `claude/commands/archive.md` (lines 1–39)

**Frontmatter:** `agent: build` (built-in; runs in the main-loop orchestrator context).

**Steps performed by the command itself:**

1. (line 20–21) If `openspec/` does not exist, stop and tell the user to run `/qrspi:init`.
2. (lines 23–27) **Sanity-check:** Read `openspec/changes/<id>/followups.md` for un-ticked `- [ ]` boxes. If any exist, surface them. This is **inform-only**, not a hard-block.
3. (lines 29–33) Load and run the `openspec-archive-change` skill, passing the change id.
4. (lines 35–36) Relay the archive summary to the user.

**No commit step.** The archive command does not call `git commit` or `git push`; the archive is a filesystem move performed by the skill. The command also does not check whether the PR has merged.

**Glob usage for discovery:** (lines 38–39) The command instructs the agent to use the Glob tool with patterns `openspec/changes/*` and `openspec/changes/archive/*` — no shell-out permitted.

### `claude/skills/openspec-archive-change/SKILL.md` (lines 1–118)

Six-step procedure:

1. **If no change name provided:** Run `openspec list --json` to enumerate active changes; prompt via AskUserQuestion. Never auto-select.
2. **Artifact completion:** Run `openspec status --change "<name>" --json`. Parses `artifacts` array for non-`done` entries. If any are incomplete, warns and asks via AskUserQuestion whether to proceed. If `actionContext.mode: "workspace-planning"` is found, stops entirely.
3. **Task completion:** Read `tasks.md`; count `- [ ]` vs `- [x]`. If incomplete tasks found, warns and asks via AskUserQuestion. If no `tasks.md`, proceeds silently.
4. **Delta spec sync assessment:** Uses `artifactPaths.specs.existingOutputPaths` from the status JSON to locate delta specs. Compares each with `openspec/specs/<capability>/spec.md`. Shows a combined summary before prompting. Prompt choices: "Sync now (recommended)" / "Archive without syncing" (or "Archive now" / "Sync anyway" / "Cancel" if already synced). If user chooses sync, spawns a subagent (via Task tool) to invoke `openspec-sync-specs`.
5. **Perform the archive:** `mkdir -p "<changesDir>/archive"` then `mv "<changeRoot>" "<changesDir>/archive/YYYY-MM-DD-<name>"`. Errors if the target already exists. The `.openspec.yaml` file moves with the directory.
6. **Display summary:** Reports change name, schema, archive location, whether specs were synced, and any warnings.

**No PR-status check anywhere in the skill.** The skill checks artifacts, tasks, and specs — it does not verify the PR has merged or even that `pr.md` exists.

The command body (lines 61–76) prescribes this exact markdown:

```markdown
# Pull request -- <id>

- **URL:** <url>
- **Title:** <id>: <summary>
```

Six fields: PR number, URL, title, source branch, target branch, created date.

The canonical shape is not uniformly followed. Archived `pr.md` files differ:

## Backlog row lifecycle

### `openspec/backlog.md` structure

- **Top-level header:** `# Backlog`
- **Section headings (`##`):** `## Ideas`, `## Proposed`, `## In progress` (implied by row placement). The file currently shows only `## Ideas` with all rows (including `archive-requires-merged-pr` as `proposed` and an `in-progress` row if present).
- **Row heading format:** `### <id> — \`<status> (<free-text note>)\`` · **`P1`|`P2`|`P3`**
- **Priority bands:** `P1` = correctness/safety, low cost; `P2` = high-value enhancements; `P3` = strategic bets or sequenced items.
- **Row body:** One-line `**Why:**` explanation (plus optional shape for complex items). No separate `Status:` or `Next QRSPI command:` field — status is entirely in the `###` heading.

### Status transition map

| Transition | Performed by | Where documented |
|---|---|---|
| (new) → `idea` row created | Q stage, D stage, or S stage (deferred-work capture) | `workflow` skill "Capturing deferred work"; `questions.md`, `design.md`, `structure.md` commands |
| `idea` → `proposed` | Q stage (questioner subagent, its step 9; verified or corrected by orchestrator) | `claude/commands/questions.md:80–83` |
| `proposed` → `in-progress (Q, R, D, S, V, P, I complete)` + move row to `## In progress` | Implement stage, final slice only | `claude/commands/implement.md:100–109` |
| `in-progress (…)` note updated to `in-progress (draft PR #<N> open)` | PR stage (main-loop orchestrator, after reviewer returns) | `claude/commands/pr.md:74–77` |
| Row **deleted** | Archive stage | Stated in `workflow` skill: "Completed work lives under `openspec/changes/archive/`, not here." and `claude/commands/pr.md:58–59` notes the row is "deleted on archive". |

**No command currently flips a row to `merged`.** The workflow skill mentions `merged` as a valid status in its prose ("Status is one of `idea` / `proposed` / `in-progress` / `merged`") but the actual backlog.md top-of-file comment also lists it; however no command body performs that transition. The archive command does not update the backlog row at all — the skill simply moves the folder.

**Atomic-commit rule:** Every backlog edit lands in the same commit as the artifact that caused the state change. This is enforced by convention (workflow skill "Backlog atomicity"), not by any lint check.

**Stages that do NOT touch the backlog row:** D (no status flip, but may add `idea` rows), S (same), V (Slices — explicitly stated not to touch), P (Plan — explicitly stated not to touch), I intermediate slices (explicitly stated not to touch), followup (post-PR fix does not change status).

## Runtime shell/tool constraints

### CLAUDE.md prohibitions (applicable to command/skill markdown)

1. **No shell-injection in slash commands** (`CLAUDE.md` section "Don't shell out in slash commands"): exclamation-mark-prefixed backtick syntax (the auto-run directive) is prohibited in `claude/commands/*.md` and `.claude/commands/*.md`. The static permission checker rejects bash redirects on Windows/PowerShell with "Unrecognized redirect shape". The prescribed alternative is the **Glob tool**.

2. **No `!`-then-backtick in prose either** (`CLAUDE.md` section "Don't write `!`-then-backtick literally"): the same scanner fires on documentation of the syntax. A placeholder like `<shell>` after `!`` triggers the redirect error. Files must split the `!` from the backtick or describe it in words.

### How commands invoke CLI at runtime (Bash tool)

Commands instruct the agent to use the **Bash tool** at runtime for git/CLI calls. This is distinct from the compile-time shell-injection prohibition: Bash tool calls are dynamic invocations inside agent turns, not static auto-run directives in the file body.

Concrete examples found in command bodies:

- `claude/commands/pr.md:22–24`: `git status --short` (Bash tool, to check working-tree cleanliness). Comment on line 24 notes it is not on the default allow-list and "the harness's obfuscation guard blocks brace+quote shapes" — so explicit path patterns are avoided.
- `claude/commands/questions.md:30–31`: `git checkout -b <branch>` and `git push -u origin <branch>` (Bash tool, as inline code blocks in the command body instructing the agent to run them).
- `claude/commands/implement.md:68–79`: `git add … && git commit … && git push` (Bash tool, per-slice commit step).
- `claude/commands/pr.md:98`: `git add openspec/changes/<id>/pr.md openspec/backlog.md openspec/changes/<id>/followups.md` (Bash tool, commit step).
- `claude/skills/openspec-archive-change/SKILL.md:76–79`, 83–86: `mkdir -p …` and `mv …` (Bash tool calls embedded as bash code blocks inside the skill).
- `claude/skills/openspec-archive-change/SKILL.md:30`, 36: `openspec list --json` and `openspec status --change "<name>" --json` (Bash tool, OpenSpec CLI calls).

The rule is: **Glob for read-only discovery; Bash tool for git/CLI operations that change state or need subprocess output**.

## Copilot sync mechanism

### `sync-copilot.mjs` — transformation overview

The script (entry point: `main()`, line 402) generates three output trees under `copilot/`:

| Source | Output directory | Count reported |
|---|---|---|
| `claude/agents/*.md` | `copilot/agents/copilot-<stem>.agent.md` | `na` |
| `claude/commands/*.md` | `copilot/prompts/qrspi-<stem>.prompt.md` | `np` |
| `claude/skills/<dir>/SKILL.md` | `copilot/instructions/<dir>.instructions.md` | `ni` |

**Command → prompt mapping for `archive.md`:**

- Source: `claude/commands/archive.md`
- Stem: `archive` (filename without `.md`)
- Output: `copilot/prompts/qrspi-archive.prompt.md` (stem is `qrspi-archive` because the script re-adds the `qrspi-` prefix at line 290)
- `agentFor` table (`sync-copilot.mjs:34–41`): `archive` is **not present** in `agentFor`, so the generated agent field becomes `agent: agent` (the generic Copilot agent, line 276).
- `hintFor` table (lines 42–48): `archive` is also not present, so no `argument-hint:` field is emitted.

**Key transformation rules (`rewriteAll`, lines 109–161):**

- `$ARGUMENTS` → `${input}` (line 117)
- `$HOME/.claude` → `$HOME/.copilot`, `~/.claude` → `~/.copilot` (lines 118–119)
- `restart Claude Code` → `reload the VS Code window` (line 120)
- Path refs: `.claude/skills/<X>/SKILL.md` → `.github/instructions/<X>.instructions.md` (line 122); `.claude/commands/<X>.md` → `.github/prompts/<X>.prompt.md` (line 123)
- Skill-load prose: `Load skill \`X\`` → `Consult the **X** instructions (\`X.instructions.md\`)` (line 135); `load the \`X\` skill` → `consult the **X** instructions` (line 136); bare `Load skills` → `Consult the instructions for` (line 137)
- Shell-injection lines: `!`` <cmd>`` ` (line format) → `Run \`<cmd>\` and use the result.` (line 139)
- `@path` lines → `#file:path` (line 140)
- `AskUserQuestion` → `#tool:vscode/askQuestions` or bare `vscode/askQuestions` (lines 147–152)
- "invoke the X subagent" → "continue as the X" (line 154)
- `/qrspi:<cmd>` → `/qrspi-<cmd>` (line 159)
- Agent references rewritten to `copilot-<role>` (line 133)

**Frontmatter generation for commands (lines 267–282):**
- Always emits `description:` (from source frontmatter).
- Emits `argument-hint:` if the stem appears in `hintFor`.
- Emits `agent: copilot-<role>` if stem in `agentFor`; otherwise `agent: agent`.
- If `agent: agent` AND body contains `AskUserQuestion`, emits `tools: [<promptToolset>]`.

**`--check` mode (lines 350–392):** Generates to a temp dir, compares file-by-file against `copilot/`, reports ADDED / DELETED / DIFF with a line-level positional diff. Exits 1 if any files differ or any `SKILL.md` was missing.

**Exclusions:** `claude/commands/qrspi-sync-copilot.md` and `claude/skills/qrspi-sync-copilot/` are skipped (lines 287, 297) so the sync-copilot tooling does not ship as a consumer-facing artifact.

## Lint / CI checks (`scripts/lint.mjs`)

Five checks run in sequence; all errors are collected before exit; exits 1 if any violation.

### Check 1: Pin agreement (lines 129–248)

Scans `claude/`, `copilot/`, `openspec/` (excluding `openspec/changes/`), `openspec-templates/`, and root-level files for occurrences of `@fission-ai/openspec@<version>` and `openspec_version: <version>`. Excludes `generatedBy:` lines in `claude/skills/openspec-*/`. Asserts all found version strings agree (does not assert a specific version, only unanimity). Fails if zero occurrences found.

**Files touched by this check:** `README.md`, `openspec/config.yaml`, `claude/skills/openspec-*/SKILL.md` (body lines only), `openspec-templates/*.template.md`, any command or skill that embeds the version in prose.

### Check 2: Frontmatter / name resolution (lines 270–403)

- **Agents** (`claude/agents/*.md`): must have `name:` and `description:`. `model:` must use alias (`opus`/`sonnet`/`haiku`), not a pinned id. `Load skill X` references must resolve to an existing `claude/skills/<X>/` directory.
- **Commands** (`claude/commands/**/*.md`, walked recursively): must have `description:`. `agent:` if present must resolve to `claude/agents/<name>.md` (or be a builtin: `build`, `agent`). `model:` same alias rule.
- **Skills** (`claude/skills/<dir>/SKILL.md`): must have `name:` and `description:`. Missing `SKILL.md` is also reported.

**Also enforced by Check 2:** Skill-reference resolution — `Load skill X` and `load the X skill` backtick-name references in agent bodies must resolve to `claude/skills/<X>/` directories (function `checkSkillRefs`, lines 369–403).

### Check 3: Heading alignment (lines 421–511)

For each OpenSpec template (`openspec-templates/*.template.md`), asserts that the corresponding agent's body contains the canonical fixed section headings:

| Template | Agent | Canonical headings checked |
|---|---|---|
| `questions.template.md` | `questioner` | 10 headings (Data model, Indexing & query performance, API, UI, Front-end state, Auth & authorization, Migrations & data, Testing, Sequencing & scope, Open product questions) |
| `design.template.md` | `designer` | 4 headings (Context, Goals / Non-Goals, Decisions, Risks / Trade-offs) |
| `proposal.template.md` | `architect` | 4 headings (Why, What Changes, Capabilities, Impact) |

| `spec-delta.template.md` | `architect` | 3 headings (ADDED Requirements, MODIFIED Requirements, REMOVED Requirements) |

**Does NOT touch command files or `README.md`.** Only agent files are read.

### Check 4: README command coverage (lines 532–569)

Bidirectional check between `claude/commands/*.md` (top-level only — `listFiles`, not `walkMd`) and `README.md`:

- **Forward:** Every `claude/commands/<stem>.md` must be referenced in `README.md` as `/qrspi:<stem>`.
- **Reverse:** Every `/qrspi:<token>` pattern in `README.md` must resolve to an existing `claude/commands/<token>.md`.

**Note:** This check uses `listFiles` (single directory, not recursive) for commands, so subdirectories of `claude/commands/` are not scanned.

### Check 5: Gate-tool / executor agreement (lines 606–639)

For every command with a non-builtin `agent:` frontmatter field, checks whether the command body reaches `AskUserQuestion` (directly or transitively via `workflow` skill choreography references). If it does, it is a violation — `AskUserQuestion` is main-loop-only and cannot run inside a subagent.

- Direct reference: body contains the string `AskUserQuestion`.
- Transitive reference: body mentions `` `workflow` `` (backtick-wrapped) AND at least one of `Stage choreography`, `commit step`, or `next-stage handoff`.

**Archive command is exempt from Check 5** because its frontmatter declares `agent: build` (a builtin), which is in `BUILTIN_AGENTS` (line 253), so the body is not checked.

## Implicit contracts and conventions

1. **Archive ≠ merge verification.** Nothing in the current archive flow checks whether the PR has merged. The `openspec-archive-change` skill does not read `pr.md`, does not call `gh`/`az`/`glab` to fetch PR status, and does not gate on merge state. The `archive.md` command body's step 2 checks only `followups.md` for un-ticked boxes.

2. **`pr.md` shape is prescribed but not enforced.** `claude/commands/pr.md:61–76` gives the exact markdown template. Neither lint nor any validation step checks that the file matches this shape. Archived examples show significant format drift.

3. **Backlog `merged` status exists in documentation but no command writes it.** The `workflow` skill's prose and `openspec/backlog.md`'s own header list `merged` as a valid status, but no command body performs the `in-progress` → `merged` transition. The row is simply deleted on archive.

4. **Archive does not update `openspec/backlog.md`.** The `openspec-archive-change` skill moves the folder; it does not remove or update the backlog row. The `workflow` skill states "remove archived rows" and `pr.md:58–59` states "this row is deleted on archive" — but neither the command nor the skill implements this deletion.

5. **No `git commit` in the archive flow.** The archive command and skill produce a filesystem move but issue no git commit. There is no stage-level commit step analogous to the QRSPI alignment stages.

6. **`openspec-archive-change` is generated, not hand-maintained.** CLAUDE.md and the command body both warn against editing it. Changes must go through OpenSpec CLI upgrades.

7. **The `agentFor` table in `sync-copilot.mjs` is a parallel representation** of the delegation declared in each command body. It maps stem names to agent roles for generating the `agent:` frontmatter of Copilot prompts. There is no automated cross-check between this table and the Agent tool invocations inside command bodies (noted in the backlog: `agentFor-frontmatter-crosscheck` — `idea` · P3).

8. **Glob for discovery, Bash tool for state-changing CLI.** This is the consistent pattern across all command bodies.

9. **`followups.md` check in archive.md is inform-only.** `archive.md:26–27` says "Inform, don't hard-block". The user may proceed to archive even with open follow-ups.

10. **`agent: build` frontmatter on `archive.md`.** This means the command runs in the main-loop orchestrator context (per Check 5 logic), not inside a named subagent. The skill is loaded and executed inline.
