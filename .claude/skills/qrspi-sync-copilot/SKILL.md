## Direction & contract

`claude/` is the **single source of truth**. `copilot/` is a **generated
artifact** -- never hand-edit it; edit `claude/` and re-run `/qrspi-sync-copilot`.
The sync overwrites `copilot/` wholesale. The OpenSpec-generated `openspec-*`
files under `claude/` are themselves regenerated by the OpenSpec CLI; we port
whatever is currently committed.

## Implementation: this spec is code

The rules below are implemented **deterministically** by `sync-copilot.mjs` (repo
root). It drop-and-recreates `copilot/`; running it twice yields the same tree
(`node sync-copilot.mjs --check` proves zero drift). Two rule layers:

- **`rewriteAll`** -- ordered text rewrites applied to every file (the
  frontmatter/path/mechanic rules below).
- **`applyFixups`** -- per-output-file literal replacements for the handful of
  cases generic rules can't express (currently `qrspi-implement` and
  `qrspi-init`).

**Golden rule: improve the script, never the generated output.** A hand-edit to a
`copilot/` file is wiped on the next run. When review (via `/qrspi-sync-copilot`)
finds a systematic gap, add a `rewriteAll` rule (broad) or an `applyFixups`
`lRep` entry (one file), then re-run. Fixes then compound and stay reproducible.

## File mapping

| Claude source | Copilot output | Notes |
|---|---|---|
| `claude/agents/<x>.md` | `copilot/agents/copilot-<x>.agent.md` | custom agent -- namespaced `copilot-` (see below) |
| `claude/commands/<x>.md` | `copilot/prompts/<x>.prompt.md` | slash prompt |
| `claude/skills/<x>/SKILL.md` | `copilot/instructions/<x>.instructions.md` | reference doc |

**Not ported:** `claude/commands/qrspi-sync-copilot.md` and this skill -- they are
Claude-side tooling; Copilot cannot run the sync.

## Frontmatter transforms

### Agent -> `.agent.md`
```yaml
---
description: <the Claude `description`, one line>
tools: [<mapped tools -- see table>]
---
```
- Drop `name` (filename is the identity) unless a display name helps; then keep `name`.
- **Namespace the filename `copilot-<x>.agent.md`.** Generated agents are
  prefixed `copilot-` so prompts/instructions reference the *generated* agent,
  never the Claude one. Body references to `.github/agents/<x>.agent.md` are
  likewise rewritten to `.github/agents/copilot-<x>.agent.md` (idempotent via a
  negative lookahead -- no `copilot-copilot-`).
- **Omit `model`.** Copilot runs the user's selected model. If the Claude agent
  was `model: opus`, add one body line near the top: `> Recommended model: a
  strong reasoning model (this stage runs on Opus under Claude Code).`

### Command -> `.prompt.md`
```yaml
---
description: <the Claude `description`, one line>
argument-hint: <hint>        # only if the command consumes $ARGUMENTS
agent: <see rule below>
---
```
- **`agent:` is the bridge for delegation.** If the Claude command delegates to a
  subagent, set `agent:` to the **namespaced** `copilot-<role>` agent so the
  prompt runs inside the generated agent (the unprefixed built-in `agent` stays
  unprefixed):

  | Command | `agent:` |
  |---|---|
  | qrspi-questions | copilot-qrspi-questioner |
  | qrspi-research | copilot-qrspi-researcher |
  | qrspi-design | copilot-qrspi-designer |
  | qrspi-structure / qrspi-worktree | copilot-qrspi-architect |
  | qrspi-plan | copilot-qrspi-planner |
  | qrspi-implement / qrspi-followup | copilot-qrspi-implementer |
  | qrspi-pr | copilot-qrspi-reviewer |
  | qrspi, qrspi-stack, qrspi-init, qrspi-archive, qrspi-retro | `agent` (no specific subagent -- runs inline) |

- Drop the Claude `agent: build` line (Claude-specific).

### Skill -> `.instructions.md`
```yaml
---
description: <the Claude `description`, one line>
---
```
- **No `applyTo`** -- these are not auto-applied; they are referenced on demand
  (see body rule). Drop `name`.

## Tool-name map (Claude -> Copilot `tools:`)

VS Code namespaced the built-in tool ids; the un-namespaced forms (`codebase`,
`editFiles`, `runCommands`) now emit "Tool or toolset 'X' has been renamed"
warnings, so the script maps to the **current namespaced ids**:

| Claude | Copilot |
|---|---|
| Read, Glob, Grep | `search`, `search/codebase` |
| Edit, Write, NotebookEdit | `edit/editFiles` |
| Bash, PowerShell | `execute/runInTerminal`, `execute/getTerminalOutput` (was the old `runCommands` toolset) |
| WebFetch, WebSearch | `web/fetch` |
| AskUserQuestion | `vscode/askQuestions` (Copilot's structured-question tool) |
| TodoWrite | *(drop -- Copilot tracks todos itself)* |
| Skill | *(drop -- replaced by `#file:` / instruction references)* |
| Agent, Task | *(drop -- replaced by the prompt's `agent:` field / handoffs)* |

De-duplicate the resulting list. `vscode/askQuestions` is in the **base** set
(every QRSPI stage has an interactive step), so it lands on every agent. A
read-only agent (e.g. researcher, reviewer) therefore gets
`[search/codebase, search, vscode/askQuestions, web/fetch]` -- no `edit/editFiles`
or `execute/*` terminal tools.

## Body-mechanic rewrites

- `$ARGUMENTS` -> `${input}`; add an `argument-hint` describing it.
- An exclamation-prefixed shell-injection line (Claude Code's auto-run-and-inject
  syntax -- do NOT reproduce the literal exclamation-then-backtick here; see
  CLAUDE.md for the static-scanner note) -> a written step: "Run `<shell>` and
  use the result." (Copilot prompts have no auto-injection.) If the line was a
  trivial existence check, fold it into prose.
- `@path` include -> `#file:path`.
- "Load skill `X`" / "load the `X` skill" -> "Consult the **X** instructions
  (`X.instructions.md`)." (User-scope instruction files aren't auto-applied, so
  this is a soft reference the model honors when the file is present.)
- "delegate to / spawn the `<x>` subagent", "via the Agent/Task tool",
  `**Model:** sonnet|opus` annotations -> "switch to the **<x>** agent". Where a
  natural next-agent exists, you may add a Copilot `handoffs: [<x>]` to the
  agent frontmatter.
- "use the **AskUserQuestion** tool" -> `#tool:vscode/askQuestions` (Copilot's
  structured-question tool -- same `#tool:` reference convention as `#file:`).
  Per-call / bare mentions -> the plain `vscode/askQuestions` id. A generic
  `agent: agent` prompt that uses it also gets an explicit `tools:` superset
  (it inherits no agent toolset); a prompt delegating to a QRSPI agent inherits
  the tool from that agent.

## Path rewrites (for install-time artifacts referenced in bodies)

- `~/.claude/` -> `~/.copilot/`
- `.claude/commands/<x>.md` -> `.github/prompts/<x>.prompt.md`
- `.claude/agents/<x>.md` -> `.github/agents/copilot-<x>.agent.md` (namespaced)
- `.claude/skills/<x>/SKILL.md` -> `.github/instructions/<x>.instructions.md`
- "restart Claude Code" -> "reload the VS Code window"

## Fidelity gaps (state these in output; do not paper over them)

1. **Per-slice model selection.** Claude picks sonnet/opus per slice and spawns
   the implementer on it. Copilot has no equivalent -- the `**Model:**` annotation
   becomes advisory prose; the human picks the model in the Copilot model picker.
2. **Automatic subagent orchestration.** Claude commands spawn subagents as
   context firewalls. Copilot runs the prompt inside one `agent:`; deeper
   delegation becomes a human-driven agent switch / `handoffs`.
3. **Skill auto-loading.** Claude loads skills on demand via the Skill tool.
   Copilot instruction files (user scope, no `applyTo`) are soft references.
4. **Shell injection & file includes** in commands degrade to written steps.
5. **Next-stage handoffs.** A QRSPI command ends by offering to "invoke
   `/qrspi-<next> <id>` now". Claude Code chains the next stage automatically;
   Copilot cannot invoke one prompt from another, so the handoff degrades to the
   model asking (via `vscode/askQuestions`) and the human running the next
   `/qrspi-<next>` prompt themselves. The text stays; only the auto-run is lost.

## After generating

Print a summary: counts per type, any artifact that needed a non-obvious
transform, and a reminder to **commit `copilot/`** (it is a generated artifact
that ships for Copilot users who never run the sync).
